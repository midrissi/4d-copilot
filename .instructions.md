### Storage Object (Application-Level Persistence)

The **Storage** object is used to store data accessible from all processes and sessions.

**⚠️ CRITICAL CONSTRAINT**: Storage properties can **ONLY** be:
- **Shared objects** (`New shared object()`)
- **Shared collections** (`New shared collection()`)
- **Null**

**You CANNOT store:**
- Primitives (Integer, Text, Boolean, etc.)
- Regular objects or collections
- Direct scalar values

**Correct usage - wrap all values in shared objects:**
```4d
// Initialize storage on first call
Use (Storage)
	If (Storage.app=Null)
		Storage.app:=New shared object(\
			"nextId"; 1; \
			"maxUsers"; 100; \
			"appName"; "MyApp")
	End if
	If (Storage.users=Null)
		Storage.users:=New shared collection
	End if
End use

// Access nested values (reading is safe without Use)
var $nextId : Integer := Storage.app.nextId
var $appName : Text := Storage.app.appName

// Modifying requires Use block
Use (Storage)
	Storage.app.nextId:=Storage.app.nextId+1
End use
```

**Incorrect usage - DO NOT DO THIS:**
```4d
// ❌ WRONG - cannot store primitive directly
Storage.nextId:=1  // Error: Integer not allowed
Storage.appName:="MyApp"  // Error: Text not allowed
Storage.isActive:=True  // Error: Boolean not allowed
```

**Best Practice**: Group related configuration in a single shared object:
```4d
// One config object for related values
Storage.app:=New shared object("nextId"; 1; "maxUsers"; 100)

// Separate shared collections for data
Storage.users:=New shared collection
Storage.sessions:=New shared collection
```

### Use Block Requirement

**⚠️ CRITICAL**: When **altering/modifying** Storage properties or collections, you MUST wrap the modification in a `Use` block with the specific object being modified.

**Why?** Storage objects and collections are shared across all processes. The `Use` block ensures thread-safe access and prevents data corruption.

**Reading is safe WITHOUT Use block:**
```4d
// ✓ Safe - reading values does NOT need Use block
var $userId : Text := Storage.users[0].id
var $nextId : Integer := Storage.app.nextId
var $count : Integer := Storage.users.length
var $found : Object := Storage.users.find(Formula($1.value.id="123"))  // ✓ Read-only, no Use needed
var $idx : Integer := Storage.users.findIndex(Formula($1.value.name="John"))  // ✓ Read-only, no Use needed
```

### ⚠️ Collection Callback: Always Use `$1.value`

When using collection iteration methods (`find`, `findIndex`, `map`, `filter`, `every`, `some`), access the current element via **`$1.value`**, not `$1` directly.

```4d
// ✓ CORRECT - use $1.value to access the current element
var $found : Object := Storage.users.find(Formula($1.value.id="123"))
var $idx : Integer := Storage.users.findIndex(Formula($1.value.name="John"))
var $ids : Collection := Storage.users.map(Formula($1.value.id))

// ❌ WRONG - $1 is the callback parameter object, not the element itself
var $found : Object := Storage.users.find(Formula($1.id="123"))  // Always null!
```

**Why**: `$1` is the callback parameter object with properties `.value` (element), `.index` (position), `.collection` (source). Accessing `.id` directly on `$1` returns null because `$1` has no `id` property.

**Modifying requires Use block with the specific object:**
```4d
// ✓ Correct - wrap with Use(Storage.object) when modifying
Use (Storage.app)
	Storage.app.nextId := Storage.app.nextId + 1  // ✓ Wrap entire app object
End use

// ✓ Correct - wrap collection modifications
Use (Storage.users)
	Storage.users.push($newUser)  // ✓ Inside Use
	Storage.users.splice($idx; 1)  // ✓ Inside Use
End use

// ✓ Correct - wrap nested property modifications
Use (Storage.users)
	Storage.users[0].name := "Updated"  // ✓ Inside Use
End use

// ✓ Correct - read-only operations do NOT need Use
var $user : Object := Storage.users.find(Formula($1.value.id="123"))  // ✓ Read, no Use needed
```

**Incorrect - modifying without Use block:**
```4d
// ❌ WRONG - modifying without Use
Storage.app.nextId := 2  // ERROR: needs Use block
Storage.users.push($item)  // ERROR: needs Use block
Storage.users[0].name := "Test"  // ERROR: needs Use block
```

**Important Rules:**
1. **Reading is always safe**: No Use block needed for reads (including `.find()`, `.findIndex()`, `.length`)
2. **Use the specific object**: `Use (Storage.app)` for app modifications, `Use (Storage.users)` for users
3. **Modifications need Use**: Property assignments, `.push()`, `.splice()`, any write operation
4. **Wrap just what you're modifying**: Be specific about which Storage property needs protection

### ⚠️ STRICT RULE: Shared Object Creation and Mutation

**When creating a new shared object and setting its properties, ALWAYS wrap the property mutations in a Use block with that object.**

**Correct pattern:**
```4d
// Step 1: Create the shared object OUTSIDE any Use block
$newUser:=New shared object:C1526()

// Step 2: Wrap ALL property mutations in Use (object)
Use ($newUser)
	$newUser.id:=String:C10($nextId)
	$newUser.name:=String:C10($body.name)
	$newUser.email:=String:C10($body.email)
End use

// Step 3: Push/store in collection with SEPARATE Use block
Use (Storage:C1525.users)
	Storage:C1525.users.push($newUser)
End use
```

**Why this matters**: 
- Properties set on shared objects must be synchronized with a Use block
- Without Use, properties may not persist when the object is accessed from other processes
- Each Use block wraps only ONE object being modified (no nesting different objects)

**CRITICAL: Three separate Use blocks are required:**
1. `Use ($newUser)` → Set properties on the shared object
2. `Use (Storage.collection)` → Add the object to storage
3. `Use (Storage.app)` → Update any counters/metadata

**DO NOT do this:**
```4d
// ❌ WRONG - properties set without Use block
$newUser:=New shared object:C1526()
$newUser.id:=String:C10($nextId)  // Properties may not persist!
Storage:C1525.users.push($newUser)

// ❌ WRONG - creating inside collection Use block
Use (Storage:C1525.users)
	$newUser:=New shared object:C1526()
	$newUser.id:=String:C10($nextId)  // Wrong: mutating without Use ($newUser)
	Storage:C1525.users.push($newUser)  // Wrong: mixing two objects in one Use
End use

// ❌ WRONG - properties set inside collection Use block (not wrapped separately)
Use (Storage:C1525.users)
	$newUser:=New shared object:C1526()
	$newUser.id:=String:C10($nextId)  // Properties may become null or undefined!
	Storage:C1525.users.push($newUser)
End use
```

**The separation is critical**: Create → Mutate (with Use) → Store (with Use)

```4d
// Log a message with optional data payload
debugSessionLog($location: Text; $message: Text; $hypothesisId: Text; $runId: Text; $payload: Object)

// Example: Log function entry
var $runId := String(Milliseconds())
debugSessionLog("handleUsers"; "POST handler called"; "hyp-001"; $runId; {method: $request.verb; bodySize: $request.getText().length})

// Log before critical operations
debugSessionLog("storage-check"; "About to store user"; "hyp-001"; $runId; {userId: $newUser.id; storage_length: Storage.users.length})

// Log errors/edge cases

	debugSessionLog("lookup-failed"; "User not found"; "hyp-001"; $runId; {searchId: $userId})
End if
```

**Output format**: JSON lines with `timestamp`, `location`, `message`, `data`, `hypothesisId`, `runId`

## Singleton Classes

Shared singleton classes are used to create a single instance accessible across all processes. ResponseHelper is a shared singleton.

**Declaration:**
```4d
shared singleton Class constructor()

Function success($result : Object) : Object
	return New object("success"; True; "result"; $result)

Function error($message : Text) : Object
	return New object("success"; False; "error"; $message)
```

**Important syntax rule:** In class files, functions do NOT use `End function`. The 4D language automatically detects the end of a function when the next `Function` keyword appears or the file ends.

**Usage:**
Access singletons via the `.me` property:
```4d
var $helper := cs.ResponseHelper.me
$helper.success($data)
```

**Why `.me` instead of `.new()`:**
- `.me` instantiates the singleton only once
- Subsequent calls return the same instance
- `.new()` would create multiple instances (wrong pattern)

For more details on singletons, refer to the [4D Classes documentation](https://developer.4d.com/docs/Concepts/classes#singleton-classes).

## JSON Response Format

### ResponseHelper Class

A shared singleton class that provides convenient methods for formatting HTTP handler responses.

**Accessing the singleton:**
```4d
var $helper := cs.ResponseHelper.me
```

**Available methods:**

- `success($result : Object) : Object` — Returns `{ success: true, result: $result }`
- `error($message : Text) : Object` — Returns `{ success: false, error: $message }`

**Usage in handlers:**

```4d
var $response : 4D.OutgoingMessage := 4D.OutgoingMessage.new()

// Success response
$response.statusCode:=200
$response.setBody(cs.ResponseHelper.me.success($userData))

// Error response  
$response.statusCode:=404
$response.setBody(cs.ResponseHelper.me.error("User not found"))
```

### Response Structure

All HTTP handler responses follow a consistent structure:

**Success Response (2xx status):**
```json
{
  "success": true,
  "result": { /* response data */ }
}
```

**Error Response (4xx/5xx status):**
```json
{
  "success": false,
  "error": "Error message describing what went wrong"
}
```

**Example handler implementation:**
```4d
Function handleUsers($request : 4D.IncomingMessage) : 4D.OutgoingMessage

	var $response : 4D.OutgoingMessage := 4D.OutgoingMessage.new()
	
	If ($request.verb="GET")
		// Success response
		$response.statusCode:=200
		$response.setBody(cs.ResponseHelper.me.success($users))
	Else
		// Error response
		$response.statusCode:=400
		$response.setBody(cs.ResponseHelper.me.error("Bad request"))
	End if
	
	return $response
End function
```

This consistent structure makes it easier for clients to:
- Reliably check `success` boolean instead of relying only on HTTP status codes
- Access data uniformly via the `result` field
- Handle errors consistently via the `error` field

## HTTP Request Handlers

### Overview

HTTP Request handlers allow you to intercept and process custom HTTP requests with your own code. When a handler intercepts a request, it bypasses standard 4D processing features and uses your custom logic instead.

**Documentation**: https://developer.4d.com/docs/WebServer/http-request-handler

### How to Set Up HTTP Handlers

Create a `HTTPHandlers.json` file in `Project/Sources/` with handler definitions:

```json
[
  {
    "class": "UsersHandling",
    "method": "handleUsers",
    "pattern": "api/users",
    "verbs": "GET,POST"
  },
  {
    "class": "UsersHandling",
    "method": "handleUserDetail",
    "pattern": "api/users/detail",
    "verbs": "GET,POST,PUT"
  }
]
```

### Handler Definition

Each handler requires:

- **"class"**: Shared singleton class name (without `cs.` prefix)
- **"method"**: Function name in the class to execute
- **"pattern"** OR **"regexPattern"**: URL pattern to match
  - `"pattern"`: Simple prefix matching (e.g., `"api/users"` matches `/api/users/...`)
  - `"regexPattern"`: Full regex pattern (e.g., `"/api/users/[0-9]+/"`)
- **"verbs"** (optional): HTTP verbs to handle (default: all verbs). Format: `"GET,POST,PUT,DELETE"` (case-insensitive)

### Important Pattern Rules

- **Order matters**: First matching pattern is executed; others are ignored
- **Most specific first**: Place detailed patterns before general ones
- **Forbidden patterns**: Cannot handle `/4DACTION`, `/rest`, `/$lib/renderer`, `/$shared`
- **Pattern matching order** is critical for deep URL hierarchies

### Example: Handler Priority Order

```json
[
  {
    "class": "InvoicesHandling",
    "method": "handleTheInvoice",
    "pattern": "docs/invoices/details/theInvoice",
    "verbs": "GET,POST"
  },
  {
    "class": "InvoicesHandling",
    "method": "handleDetails",
    "pattern": "docs/invoices/details",
    "verbs": "GET"
  },
  {
    "class": "InvoicesHandling",
    "method": "handleInvoices",
    "pattern": "docs/invoices",
    "verbs": "GET"
  }
]
```

### Handler Function Structure

Handler functions must be in a **shared singleton class**:

```4d
shared singleton Class UsersHandling

Function handleUsers($request : 4D.IncomingMessage) : 4D.OutgoingMessage
	var $response : 4D.OutgoingMessage
	var $body : Text
	
	$response := 4D.OutgoingMessage.new()
	
	// Access request data
	var $verb : Text := $request.verb  // GET, POST, etc.
	var $url : Text := $request.url
	var $urlQuery : Object := $request.urlQuery  // URL parameters
	var $headers : Object := $request.headers
	var $bodyText : Text := $request.getText()
	var $bodyJSON : Object := $request.getJSON()
	var $blob : Blob := $request.getBlob()
	
	// Process request
	Case of
		: ($verb = "GET")
			$body := "List of users"
		: ($verb = "POST")
			$body := "User created"
		: ($verb = "PUT")
			$body := "User updated"
	End case
	
	// Set response
	$response.setBody($body)
	$response.setHeader("Content-Type"; "application/json")
	$response.setStatus(200; "OK")
	
	return $response
End function
```

### IncomingMessage Class Properties

- `.verb` - HTTP verb (GET, POST, PUT, DELETE, etc.)
- `.url` - Full request URL
- `.urlPath` - URL path without domain
- `.urlQuery` - Query parameters as object
- `.headers` - Request headers as object
- `.getText()` - Get body as text
- `.getJSON()` - Get body as JSON object
- `.getBlob()` - Get body as binary
- `.getHeader(name)` - Get specific header value

### OutgoingMessage Class Methods

- `.setBody(content)` - Set response body. Automatically sets Content-Type header:
  - Object/Collection → `application/json` (no need for JSON Stringify)
  - Text → `text/plain`
  - Blob → `application/octet-stream`
- `.setStatus(code; message)` - Set HTTP status (e.g., 200, 404, 500)
- `.setHeader(name; value)` - Set response header
- `.addHeader(name; value)` - Add response header

**Example - automatic Content-Type handling:**
```4d
var $users : Collection := Storage.users
$response.setBody($users)  // ✓ Automatically sets Content-Type: application/json

var $jsonText : Text := JSON Stringify($users)
$response.setBody($jsonText)  // Sets Content-Type: text/plain (wrong!)

// Always pass the object/collection directly for proper JSON content type
$response.setBody(New object("id"; 1; "name"; "John"))  // ✓ Correct
```

### Authentication and Privileges

- HTTP handlers are called from standard web requests (not REST)
- They may execute without prior authentication (guest session)
- In restricted mode, ensure guest privileges allow handler execution
- Handler functions can check session privileges to control access

Example:
```4d
// Check if user has permission
If (Session.getPrivileges().has("admin"))
	// Admin code
Else
	$response.setStatus(403; "Forbidden")
	$response.setBody("Access denied")
End if
```

### Notes

- Restart the web server for HTTPHandlers.json changes to take effect
- Handler functions are not exposed to external REST calls
- Most specific URL patterns should be defined first
- Only one handler per request is executed

## 4D Best Practices

1. **Error Handling**: Use `THROW()` for error conditions
2. **Type Safety**: Always declare parameter and return types with `#DECLARE`
3. **Documentation**: Include `//%attributes` metadata on all methods
4. **Parameter Separator**: Always use semicolon (;) between parameters
5. **Variable Naming**: Use descriptive names with $ prefix for local variables
6. **Code Structure**: Keep methods focused on single responsibilities

## Key Differences from Other Languages

- Uses semicolon (;) NOT comma (,) for parameter separation
- Variable assignment uses `:=` NOT `=`
- Comparison uses `=` NOT `==`
- Command names are case-insensitive but conventionally capitalized
- No curly braces; uses `End if`, `End for`, etc.

## When Writing 4D Code

1. Check official docs for accurate command signatures at https://developer.4d.com/docs/category/themes
2. Use correct semicolon-separated syntax
3. Always use `$` prefix for variables
4. Include method metadata with `//%attributes`
5. Respect 4D type system
6. Use proper error handling with `THROW()`
